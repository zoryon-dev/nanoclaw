# NanoClaw v2 — Plan 2.6: Cron Auto-Respawn + Admin Command Fixes

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fix 3 pre-existing bugs in shared NanoClaw v2 code: admin commands always denied via every channel (Bug A), cron auto-respawn broken (Bug B), and ISO timestamps that never compare as due in SQLite (Bug C).

**Architecture:** Three layers of fix, all at the consumer side (no adapter churn):
1. `container/agent-runner/src/formatter.ts` composes `senderId` with channel prefix (`telegram:`, `whatsapp:`) and swarm-suffix stripping.
2. `src/host-sweep.ts` makes `handleRecurrence` sync (top-level cron-parser import, drop async, drop await).
3. New shared helper `src/db/sqlite-utc.ts` replaces both the inline `toSqliteUtc` in `register-cron-jobs.ts` and the broken `.toISOString()` in `host-sweep.ts`.

**Tech Stack:** TypeScript + vitest (NanoClaw v2), `better-sqlite3` against `:memory:` for tests, `cron-parser` (already a dep), Docker (container rebuild).

**Spec:** [docs/superpowers/specs/2026-05-12-cron-respawn-and-admin-fixes-design.md](../specs/2026-05-12-cron-respawn-and-admin-fixes-design.md) (commit `a907138`).

## Naming and conventions

- **Helper signature:** `toSqliteUtc(d: Date): string` — returns `'YYYY-MM-DD HH:MM:SS'` UTC.
- **Channel prefix conventions** (already in use by `router.ts:332`): `whatsapp:`, `telegram:`, `discord:`, etc. The userKind is `channel_type.split('-')[0]` to strip swarm suffixes (`telegram-finance` → `telegram`).
- **Already-prefixed senderId**: passes through unchanged (defensive — if a future adapter populates `content.senderId='telegram:xyz'` directly, don't double-prefix).
- **Existing test patterns**: `formatter.test.ts` uses a `row(content)` helper for fixture `MessageInRow` rows. Reuse this helper. Vitest uses `describe`/`it`/`expect` from `vitest`.
- **DB tests**: in-memory or temp-file SQLite via `better-sqlite3`. See `scripts/finance/__tests__/register-cron-jobs.test.ts` for the established pattern (mkdtemp + schema exec + close + reopen for assertions).

```typescript
/**
 * Format a Date as 'YYYY-MM-DD HH:MM:SS' in UTC — matches the output of
 * SQLite's datetime('now') so string comparisons like
 *   WHERE process_after <= datetime('now')
 * compare correctly. Do NOT use Date.toISOString() for any TEXT column you
 * intend to compare against datetime('now'): 'T' > ' ' in ASCII, so an ISO
 * timestamp never satisfies the predicate until well after its intended time.
 * See commit 1c20a71 (Plan 2 fix in scripts/finance/register-cron-jobs.ts)
 * and Plan 2.6 spec §2 (Bug C) for full context.
 */
export function toSqliteUtc(d: Date): string {
  return d.toISOString().slice(0, 19).replace('T', ' ');
}
```

```typescript
export function categorizeMessage(msg: MessageInRow): CommandInfo {
  const content = parseContent(msg.content);
  const text = (content.text || '').trim();
  // Compose senderId with channel prefix so admin checks match the prefixed
  // entries in NANOCLAW_ADMIN_USER_IDS (e.g. 'telegram:8557164566').
  // Strip swarm suffix (e.g. 'telegram-finance' → 'telegram') the same way
  // router.ts:extractAndUpsertUser does. WhatsApp native adapters set
  // content.sender; Chat SDK bridge sets content.author.userId; either form
  // (or an already-prefixed content.senderId) works.
  const rawSender =
    (typeof content.senderId === 'string' && content.senderId) ||
    (typeof content.author?.userId === 'string' && content.author.userId) ||
    (typeof content.sender === 'string' && content.sender) ||
    null;
  const userKind = (msg.channel_type || '').split('-')[0];
  const senderId =
    rawSender === null
      ? null
      : rawSender.includes(':') || !userKind
        ? rawSender
        : `${userKind}:${rawSender}`;
```

```typescript
    // is the correct behavior for "not yet due".
    const futureDue = db
      .prepare("SELECT id FROM messages_in WHERE process_after <= datetime('now') AND id = ?")
      .get(respawn.id);

    // And if we manipulate it backward to a known past time and re-check,
    // the same query MUST return the row (proves the comparison works).
    db.prepare("UPDATE messages_in SET process_after = '2026-01-01 00:00:00' WHERE id = ?").run(respawn.id);
    const pastDue = db
      .prepare("SELECT id FROM messages_in WHERE process_after <= datetime('now') AND id = ?")
      .get(respawn.id);
    expect(pastDue).toBeDefined();
  });
});
```

```typescript
export function handleRecurrence(inDb: Database.Database, session: Session): void {
```
